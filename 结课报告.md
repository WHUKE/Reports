## 2 知识图谱构建

### 2.1 数据源、获取方法和原始数据统计信息

本项目的数据源主要来自国内主流企业公开招聘网站，包括腾讯招聘、百度招聘、美团招聘和京东招聘。为了提升岗位样本的代表性与时效性，实现了面向多站点的岗位爬虫模块，自动抓取公开岗位信息并统一整理为原始文本数据，作为后续实体抽取与知识图谱构建的输入。

**数据源与获取范围。**  
项目当前接入了 4 个招聘站点：腾讯、百度、美团、京东。各站点均优先抓取技术类岗位，以保证数据主题与“职业技能知识图谱”建设目标一致。具体来说，腾讯站点通过岗位类别参数 `parentCategoryId=40001` 过滤技术岗位；百度站点通过 `postType=1` 和 `recruitType=SOCIAL` 抓取社会招聘中的技术岗位；美团站点通过 `jfJgList` 与 `jobType` 等筛选参数限定技术岗位；京东站点则通过 `jobTypeJson=["YANFA"]` 聚焦研发类岗位。

从抓取流程上看，各站点遵循统一的处理链路：  
1. 按页调用岗位列表接口，获取岗位概要信息；  
2. 从列表结果中提取岗位 ID；  
3. 若站点提供独立详情接口，则继续根据岗位 ID 请求详情数据；  
4. 将不同站点返回的异构字段统一映射为内部的 `RawJobPosting` 对象；  
5. 再格式化为统一风格的原始文本 `.txt` 文件写入 `data/raw/<site>/`；  
6. 同时将抓取状态、岗位 ID、来源 URL、文件名、抓取时间、页码等信息追加记录到 `manifest.jsonl`，便于后续追踪、去重和失败重试。

其中，腾讯和美团采用“列表接口 + 详情接口”的两段式抓取方式：先通过列表接口获取岗位 `PostId` 或 `jobUnionId`，再请求详情接口获得完整 JD 内容；百度和京东的列表接口已返回较完整的岗位字段，因此可以直接将列表结果整理为原始文本并落盘。该设计减少了站点差异对后续解析模块的影响，使下游只需要面对统一格式的原始 JD 文本。

### 2.2 实体抽取

#### 2.2.2 基于LLM的深度知识抽取与混合解析策略（谢羿衡）

为提高技能实体抽取的细粒度和准确性，本项目设计并实现了基于DeepSeek LLM的深度知识抽取通道，作为正则解析的增强方案。

**混合解析架构**

采用"正则预解析 + LLM深度抽取"的两阶段混合策略：

1. **阶段一：正则基础解析**  
   RegexParser 先快速提取结构化元数据（职位名称、地点、学历、工作年限、部门等），作为兜底结果。

2. **阶段二：LLM深度抽取**  
   LLMParser 调用 DeepSeek API 进行细粒度技能拆分、职责结构化和语义理解，将结果与阶段一合并。

**LLM系统提示词设计**

设计了88行的结构化系统提示词（见 [config.py:11-87](https://github.com/WHUKE/JDParser/blob/main/src/core/config.py#L11-L87)），包含：

- **JSON Schema定义**：明确17个输出字段的格式和语义
- **技能抽取10条规则**：
  - 最小粒度拆分（如"MySQL/Redis/MongoDB" → 3个独立技能）
  - 必填字段约束（name, proficiency, category, skill_type, parent）
  - 技能类型分类（hard/soft/domain/certification/language/tool）
  - 层级关系识别（parent字段，如Helm的父技能为Kubernetes）
- **字段推断规则**：job_level（根据年限和职责推断）、salary解析等
- **输出严控**：禁止schema外字段，熟练度等枚举值强制映射

**实现细节**（见 [llm_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/llm_parser.py)）

```python
# 核心逻辑
def parse(self, text: str, filename: str) -> JobDescription:
    # 1. 正则解析器提取基础字段（兜底）
    jd = self._regex_parser.parse(text, filename)
    
    # 2. LLM深度抽取
    llm_result = self._call_llm(text)
    if llm_result:
        self.merge_extracted_result(jd, llm_result)
    
    return jd
```

**工程化处理**：
- **速率限制**：请求间隔0.5秒，避免触发API限流
- **错误重试**：最多3次重试，指数退避延迟
- **JSON解析容错**：处理LLM返回格式异常，降级到正则结果

**Langbase Workflow 集成**（网易LLM网关）

在网易实习期间，额外实现了基于公司内部 Langbase 平台的解析通道（见 [langbase_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py)，328行）。Langbase 采用 Workflow 异步执行模式，需要更复杂的工程化处理：

1. **异步 Workflow 模式**
   - 先调用 `trigger` 接口启动任务，获取 `runID`
   - 再轮询 `workflow-runs` 接口直到任务完成（最多60轮，每轮1秒）

2. **批量并发处理**（见 [langbase_parser.py:63-123](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L63-L123)）
   - 每批并发触发5个任务（`LANGBASE_BATCH_SIZE`）
   - 批量轮询减少API调用次数，提升吞吐量
   - 保持输入输出顺序一致性

3. **智能限流重试**（见 [langbase_parser.py:200-237](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L200-L237)）
   - 触发阶段：检测 HTTP 429，使用 `Retry-After` 头指定的延迟
   - 执行阶段：解析 workflow `failed` 消息，识别上游限流（`"429" in message`）
   - 限流失败时自动重新触发任务（最多3次），避免整批失败

4. **请求速率控制**
   - 每次请求间隔1秒（`LANGBASE_REQUEST_INTERVAL`）
   - 避免短时间内大量请求触发平台限流

**批处理流程示例**：

```python
# pipeline.py 中的批处理调度
if isinstance(self._parser, LangbaseParser):
    results = self._process_directory_batch(txt_files, output_dir)
    
# 分3步完成一批任务
# Step 1: 批量触发（5个并发）
for text, filename in batch_items:
    run_info = self._trigger(text)
    tasks.append(TaskInfo(...))
    
# Step 2: 批量轮询（多任务并行等待）
while pending_tasks:
    for task in pending_tasks:
        status = check_status(task.run_id)
        if status == "success": merge_result()
        elif status == "failed" and is_rate_limit: retry()
        
# Step 3: 收集结果并写入文件
```

**效果对比**：

| 解析模式 | 请求方式 | 处理50个JD耗时 | 限流处理 |
|---------|---------|--------------|---------|
| LLM (DeepSeek) | 同步API调用 | ~30秒（0.5s间隔） | 指数退避重试 |
| Langbase | 异步Workflow + 批处理 | ~80秒（包含轮询） | 智能识别 + 自动重触发 |

Langbase 模式虽然单次耗时更长（异步轮询开销），但支持批量并发和细粒度限流处理，适合大规模数据解析场景。

**效果**

相比纯正则解析，LLM模式能够：
- 识别隐含技能（如"分布式架构经验" → 提取"分布式系统""微服务架构"等）
- 准确区分必需技能与加分技能
- 推断技能熟练度和层级关系
- 结构化拆解职责描述

### 2.3 关系抽取

#### 2.3.1 技能归一化与知识融合（谢羿衡）

为确保同一技能在知识图谱中只对应唯一节点，避免别名导致的实体冗余和关系混乱，开发了完整的技能归一化模块（见 [normalizer.py](D:\Code\WHUKE\JDParser\src\core\normalizer.py)，465行代码）。

**归一化规则库**

构建了三层归一化规则体系（见 [normalizer.py](https://github.com/WHUKE/JDParser/blob/main/src/core/normalizer.py)）：

1. **技能别名映射表**（168条规则）  
   将各种缩写、俚语、版本号统一到规范名称：
   - 编程语言：`js/es6/es6+` → `JavaScript`，`k8s` → `Kubernetes`
   - 框架：`vue2/vue3/vue.js` → `Vue.js`，`springboot` → `Spring Boot`
   - 数据库：`mongo` → `MongoDB`，`postgres` → `PostgreSQL`
   - 云平台：`阿里云/aliyun` → `阿里云`，`gcp` → `Google Cloud`
   - 概念：`微服务` → `微服务架构`，`分布式` → `分布式系统`

2. **类别纠错映射表**（131条规则）  
   修正LLM可能误分类到"其他"的技能，补充正确的category和skill_type：
   ```python
   # 示例规则
   "html": ("编程语言", "hard"),
   "webpack": ("构建工具", "hard"),
   "docker": ("容器与编排", "hard"),
   "沟通能力": ("软技能", "soft"),
   "cet-6": ("通用技能", "certification"),
   ```

3. **熟练度数值映射**  
   将文本熟练度标准化为可比较的数值排名：
   ```
   了解=1, 熟悉=2, 熟练=3, 精通=4, 不限/其他=None
   ```

**归一化流程**（见 `normalize_skills()` 函数）

```python
def normalize_skills(skills: list[Skill]) -> list[Skill]:
    for skill in skills:
        # 1. 名称归一化（别名 → 规范名）
        skill.name = normalize_skill_name(skill.name)
        
        # 2. 类别纠错（修正category和skill_type）
        fix_skill_category(skill)
        
        # 3. 熟练度映射（文本 → 数值）
        skill.proficiency_rank = compute_proficiency_rank(skill.proficiency)
        
        # 4. 去重（同名技能只保留首次出现）
    return result
```

**工程价值**

- **减少图谱冗余**：168条别名规则确保"JavaScript"不会因为"js""JS""es6"等写法产生4个节点
- **提高查询召回**：用户搜索"K8S"时能关联到所有标记为"Kubernetes"的岗位
- **支持技能聚合**：归一化后的`proficiency_rank`可直接用于技能熟练度统计和排序

**扩展性设计**

规则表采用字典结构，新增别名只需添加一行映射，无需修改代码逻辑。类别纠错表独立维护，可持续迭代优化LLM分类误差。
