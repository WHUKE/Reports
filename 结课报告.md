## 2 知识图谱构建

### 2.1 数据源、获取方法和原始数据统计信息 (李嘉轩)

本项目的数据源主要来自国内主流企业公开招聘网站，包括腾讯招聘、百度招聘、美团招聘和京东招聘。为了提升岗位样本的代表性与时效性，实现了面向多站点的岗位爬虫模块，自动抓取公开岗位信息并统一整理为原始文本数据，作为后续实体抽取与知识图谱构建的输入。

**数据源与获取范围。**  
项目当前接入了 4 个招聘站点：腾讯、百度、美团、京东。各站点均优先抓取技术类岗位，以保证数据主题与“职业技能知识图谱”建设目标一致。具体来说，腾讯站点通过岗位类别参数 `parentCategoryId=40001` 过滤技术岗位；百度站点通过 `postType=1` 和 `recruitType=SOCIAL` 抓取社会招聘中的技术岗位；美团站点通过 `jfJgList` 与 `jobType` 等筛选参数限定技术岗位；京东站点则通过 `jobTypeJson=["YANFA"]` 聚焦研发类岗位。

从抓取流程上看，各站点遵循统一的处理链路：  
1. 按页调用岗位列表接口，获取岗位概要信息；  
2. 从列表结果中提取岗位 ID；  
3. 若站点提供独立详情接口，则继续根据岗位 ID 请求详情数据；  
4. 将不同站点返回的异构字段统一映射为内部的 `RawJobPosting` 对象；  
5. 再格式化为统一风格的原始文本 `.txt` 文件写入 `data/raw/<site>/`；  
6. 同时将抓取状态、岗位 ID、来源 URL、文件名、抓取时间、页码等信息追加记录到 `manifest.jsonl`，便于后续追踪、去重和失败重试。

其中，腾讯和美团采用“列表接口 + 详情接口”的两段式抓取方式：先通过列表接口获取岗位 `PostId` 或 `jobUnionId`，再请求详情接口获得完整 JD 内容；百度和京东的列表接口已返回较完整的岗位字段，因此可以直接将列表结果整理为原始文本并落盘。该设计减少了站点差异对后续解析模块的影响，使下游只需要面对统一格式的原始 JD 文本。

#### 2.2.2 基于LLM的深度知识抽取与混合解析策略（谢羿衡）

为提高技能实体抽取的细粒度和准确性，本项目设计并实现了基于DeepSeek LLM的深度知识抽取通道，作为正则解析的增强方案。

**混合解析架构**

采用"正则预解析 + LLM深度抽取"的两阶段混合策略：

1. **阶段一：正则基础解析**  
   RegexParser 先快速提取结构化元数据（职位名称、地点、学历、工作年限、部门等），作为兜底结果。

2. **阶段二：LLM深度抽取**  
   LLMParser 调用 DeepSeek API 进行细粒度技能拆分、职责结构化和语义理解，将结果与阶段一合并。

**LLM系统提示词设计**

设计了88行的结构化系统提示词（见 [config.py:11-87](https://github.com/WHUKE/JDParser/blob/main/src/core/config.py#L11-L87)），包含：

- **JSON Schema定义**：明确17个输出字段的格式和语义
- **技能抽取10条规则**：
  - 最小粒度拆分（如"MySQL/Redis/MongoDB" → 3个独立技能）
  - 必填字段约束（name, proficiency, category, skill_type, parent）
  - 技能类型分类（hard/soft/domain/certification/language/tool）
  - 层级关系识别（parent字段，如Helm的父技能为Kubernetes）
- **字段推断规则**：job_level（根据年限和职责推断）、salary解析等
- **输出严控**：禁止schema外字段，熟练度等枚举值强制映射

**实现细节**（见 [llm_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/llm_parser.py)）

```python
# 核心逻辑
def parse(self, text: str, filename: str) -> JobDescription:
    # 1. 正则解析器提取基础字段（兜底）
    jd = self._regex_parser.parse(text, filename)
    
    # 2. LLM深度抽取
    llm_result = self._call_llm(text)
    if llm_result:
        self.merge_extracted_result(jd, llm_result)
    
    return jd
```

**工程化处理**：
- **速率限制**：请求间隔0.5秒，避免触发API限流
- **错误重试**：最多3次重试，指数退避延迟
- **JSON解析容错**：处理LLM返回格式异常，降级到正则结果

**Langbase Workflow 集成**（网易LLM网关）

在网易实习期间，额外实现了基于公司内部 Langbase 平台的解析通道（见 [langbase_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py)，328行）。Langbase 采用 Workflow 异步执行模式，需要更复杂的工程化处理：

1. **异步 Workflow 模式**
   - 先调用 `trigger` 接口启动任务，获取 `runID`
   - 再轮询 `workflow-runs` 接口直到任务完成（最多60轮，每轮1秒）

2. **批量并发处理**（见 [langbase_parser.py:63-123](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L63-L123)）
   - 每批并发触发5个任务（`LANGBASE_BATCH_SIZE`）
   - 批量轮询减少API调用次数，提升吞吐量
   - 保持输入输出顺序一致性

3. **智能限流重试**（见 [langbase_parser.py:200-237](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L200-L237)）
   - 触发阶段：检测 HTTP 429，使用 `Retry-After` 头指定的延迟
   - 执行阶段：解析 workflow `failed` 消息，识别上游限流（`"429" in message`）
   - 限流失败时自动重新触发任务（最多3次），避免整批失败

4. **请求速率控制**
   - 每次请求间隔1秒（`LANGBASE_REQUEST_INTERVAL`）
   - 避免短时间内大量请求触发平台限流

**批处理流程示例**：

```python
# pipeline.py 中的批处理调度
if isinstance(self._parser, LangbaseParser):
    results = self._process_directory_batch(txt_files, output_dir)
    
# 分3步完成一批任务
# Step 1: 批量触发（5个并发）
for text, filename in batch_items:
    run_info = self._trigger(text)
    tasks.append(TaskInfo(...))
    
# Step 2: 批量轮询（多任务并行等待）
while pending_tasks:
    for task in pending_tasks:
        status = check_status(task.run_id)
        if status == "success": merge_result()
        elif status == "failed" and is_rate_limit: retry()
        
# Step 3: 收集结果并写入文件
```

**效果对比**：

| 解析模式 | 请求方式 | 处理50个JD耗时 | 限流处理 |
|---------|---------|--------------|---------|
| LLM (DeepSeek) | 同步API调用 | ~30秒（0.5s间隔） | 指数退避重试 |
| Langbase | 异步Workflow + 批处理 | ~80秒（包含轮询） | 智能识别 + 自动重触发 |

Langbase 模式虽然单次耗时更长（异步轮询开销），但支持批量并发和细粒度限流处理，适合大规模数据解析场景。

**效果**

相比纯正则解析，LLM模式能够：
- 识别隐含技能（如"分布式架构经验" → 提取"分布式系统""微服务架构"等）
- 准确区分必需技能与加分技能
- 推断技能熟练度和层级关系
- 结构化拆解职责描述

### 2.3 关系抽取

#### 2.3.1 技能归一化与知识融合（谢羿衡）

为确保同一技能在知识图谱中只对应唯一节点，避免别名导致的实体冗余和关系混乱，开发了完整的技能归一化模块（见 [normalizer.py](D:\Code\WHUKE\JDParser\src\core\normalizer.py)，465行代码）。

**归一化规则库**

构建了三层归一化规则体系（见 [normalizer.py](https://github.com/WHUKE/JDParser/blob/main/src/core/normalizer.py)）：

1. **技能别名映射表**（168条规则）  
   将各种缩写、俚语、版本号统一到规范名称：
   - 编程语言：`js/es6/es6+` → `JavaScript`，`k8s` → `Kubernetes`
   - 框架：`vue2/vue3/vue.js` → `Vue.js`，`springboot` → `Spring Boot`
   - 数据库：`mongo` → `MongoDB`，`postgres` → `PostgreSQL`
   - 云平台：`阿里云/aliyun` → `阿里云`，`gcp` → `Google Cloud`
   - 概念：`微服务` → `微服务架构`，`分布式` → `分布式系统`

2. **类别纠错映射表**（131条规则）  
   修正LLM可能误分类到"其他"的技能，补充正确的category和skill_type：
   ```python
   # 示例规则
   "html": ("编程语言", "hard"),
   "webpack": ("构建工具", "hard"),
   "docker": ("容器与编排", "hard"),
   "沟通能力": ("软技能", "soft"),
   "cet-6": ("通用技能", "certification"),
   ```

3. **熟练度数值映射**  
   将文本熟练度标准化为可比较的数值排名：
   ```
   了解=1, 熟悉=2, 熟练=3, 精通=4, 不限/其他=None
   ```

**归一化流程**（见 `normalize_skills()` 函数）

```python
def normalize_skills(skills: list[Skill]) -> list[Skill]:
    for skill in skills:
        # 1. 名称归一化（别名 → 规范名）
        skill.name = normalize_skill_name(skill.name)
        
        # 2. 类别纠错（修正category和skill_type）
        fix_skill_category(skill)
        
        # 3. 熟练度映射（文本 → 数值）
        skill.proficiency_rank = compute_proficiency_rank(skill.proficiency)
        
        # 4. 去重（同名技能只保留首次出现）
    return result
```

**工程价值**

- **减少图谱冗余**：168条别名规则确保"JavaScript"不会因为"js""JS""es6"等写法产生4个节点
- **提高查询召回**：用户搜索"K8S"时能关联到所有标记为"Kubernetes"的岗位
- **支持技能聚合**：归一化后的`proficiency_rank`可直接用于技能熟练度统计和排序

**扩展性设计**

规则表采用字典结构，新增别名只需添加一行映射，无需修改代码逻辑。类别纠错表独立维护，可持续迭代优化LLM分类误差。

### 2.5 知识图谱的规模、存储和可视化结果

#### 2.5.1 图谱可视化数据层与后端接口（邢泽雷）

本模块的后端图谱聚合逻辑位于 `JDGraphBuilder` 目录下，主要涵盖了图查询层与 API 接口服务层（源码见 [graph_query.py](../JDGraphBuilder/src/query/graph_query.py) 和 [serve.py](../JDGraphBuilder/src/cli/serve.py)）。其主要功能是针对图谱中繁杂的节点网络和海量边界进行实时聚合与图遍历降采样提取，从而为前端应用提供图谱可视化（如 D3.js 渲染所需的力导向基础节点关系）所需的子图结构。目前这一部分的代码已被删除，采用了云端部署的方式。

**核心子图网络聚合逻辑（`get_visual_graph`）**：
1. **起手种子选择与图关联发散**：将前端传入的检索关键词 `focus_skill`（如"Kubernetes"）作为网络中心种子节点发散；若无查询词，默认调取全库内排名前列的热门核心关键技能节点作为图网络大盘展示。
2. **关系维度网发散图层**：
   - 首先利用 `get_related_skills` 将与目标技能构建强共现且协同度高的周边技能节点（关联边种类为 `CO_OCCURS_WITH`）拉取，组装至返回数据网。
   - 随后再调起 `find_jobs_by_skills` 方法寻找与技能高度绑定的工作岗位群（`role` 型节点），并且以岗位为新介点跨维度辐射出关联地点（`location`）、学历属性要求（`education`）、岗位方向分类属性（`category`），这些属性分别映射为图模型中 `LOCATED_IN`、`REQUIRES_EDUCATION` 等具象化结构关联边。
3. **前端渲染友好输出适配**：该模块聚合得到的图结果数据中，为了保证前显不会呈现重复组件，内部利用节点特征名 ID 主键实现了映射去重。在响应组装时，将其转化为高度友好的包含 `{ "nodes": [...], "edges": [...] }` 标准字典数组格式，同时携带包含了当前连接边点统计总量的 `meta` 特量结构返回，最小化减轻前端组件二次解包的数据处理代价。

**轻量化接口引擎设计**：
使用原生无依赖的 `http.server.ThreadingHTTPServer` 搭建了名为 `JDGraphBuilderAPI` 的纯净微后端网络架构：
- 设计并接管暴露下端的 `/api/graph` 结构作为可视化功能引擎服务端点，能实时读取并在层接结构时利用参数（如把拉取发散规模控制的 `related_skill_limit` 值 与 岗位密度极限 `jobs_per_skill`）过滤数据，确保图网络信息量不至发生前台雪崩。
- 配置了包含 HTTP 头对跨域 `CORS` 等交互许可规则支持机制，同时为系统添加了微路由拦截模块：以满足分发 `/front/*` 以及全根路径的静止资产响应拦截，达到服务网端可以直接实现挂载渲染和托管本 Web HTML 项目轻量化全息发出的能力。

#### 2.5.2 可视化交互面板与前端界面设计（邢泽雷）

前端界面与系统的交互承接端为自行组建并构想布局命名的产品页—— **Woodle AI 知识探索端**（代码存放于项目下的 [woodle_ai_home.html](../JDGraphMono/front/woodle_ai_home.html)）。本页体使用免去包组件依赖特性的纯极简单文件驱动（HTML框架辅搭配 CSS 艺术结构）构造出犹如当代无瑕搜索引擎的高定交互质感。

**界面美学形态拟态流体设计特质**：
1. **复刻引擎核心布局结构感**：核心置中了具有彩色四种基源色拟形搭配的重绘 Logo ("Woodle")。搭配拥有巨大内距及输入提示的胶囊椭圆化主控搜索域（`.search-box` 的 CSS 应用流体特征），赋予产品本身干净及初见的通透空间感。
2. **高磨砂玻璃拟态组件阴影特效**：为了弱化图框死板特征，背景通过采用 `radial-gradient` 内扩型网格流质混合渐变体系以及利用包含玻璃透白遮罩（复合 `backdrop-filter: none` 或高阶透窗控制层）在视差中产生清透立体悬浮关系。例如聚焦反馈或系统快捷推荐点击时（芯片流 `.chip` 组件模块），引入触发伴随 140ms ~ 160毫秒过渡时间动画（`transition: transform` 与重影阴影），赋予物理悬空级向上抬托运动（`transform: translateY(-1px)`），大幅强化操作回弹力反馈感及系统流畅阻尼响应表现。
3. **系统视图无感态解耦结构变型架构**：基于轻量化构建宗旨，未采用三方路由。将页面分装为第一交互承接态的搜索引擎体及深延拓展使用的宽幅全景图探针结构渲染阵地面体（`.graph-panel`）两大区块。在需要发生转进流程时，由 CSS原生 `play/none` 特殊选择器类态 `.active` 的推加，隐式处理状态过渡流向操作无缝展示。

**定制全端级图可视化底座（Graph Workspace Panel）**：
知识网络进行查询唤起后的大尺寸视野控制操作版展示：
1. **多向交互图动态操作杆集**：面板顶部具备对引擎的调节条阵列（`.control-input` 控件），用来方便探索者直接使用拉杆等数值修改，实时影响触发查询传导规模及发散强度的极限设定调节；外加侧方提供了多组命令交互功能。
2. **沉浸环境底图区**：在内部留有的挂载全屏力引向渲染图及 SVG 对象图的大画布外，为其搭建防重叠隔离。在其周边提供（`.graph-status`），可以执行例如监测返回时间及因为错误超时引发红牌断供等展示容灾及友好的弹性回落降级状态报错体系体验。

**个人工作总结**

本人在图谱可视化实现与前端交互设计模块中主要负责以下工作：

1. **构建图可视化数据引擎与服务端接口**
   - 设计并实现了图谱分析结果到前端可视化组件的数据转译层（`graph_query.py`），提供基于种子扩展的强降采样与节点过滤策略（`get_visual_graph`），防止大型连通图引起前端雪崩。
   - 脱离重量级第三方 Web 框架依赖，完全基于 Python 自带的 `http.server` 模块，搭建了包含资产托管拦截、`CORS` 跨域处理与微路由在内的轻量 API 服务（`serve.py`）。
2. **Woodle AI 前端交互与纯原生 SPA 底座开发**
   - 独立构思并设计了 “Woodle AI” 极简高态搜索引擎级视图交互，使用了原生 HTML 与流体 CSS 动画（如利用 `backdrop-filter` 和原生 `radial-gradient` 呈现高磨砂拟态空间悬浮体系）。
   - 实现无三方路由下纯靠 CSS 选择器控制的视图平滑转场，集成动态调节配置栏台和错误自动收档重试体验的力导向渲染探针面板组件。

**技术亮点**：
- **无依赖轻量服务端架构**：未依赖 Flask / FastAPI 等大型全家桶框架，借助原生库构建稳定轻便的微后置接口处理和基于 JSON 的纯净 API 网关及静态文件映射器。
- **性能抗载聚合限流**：针对力导向图可视化数据爆炸瓶颈，使用后端过滤节点共现度（`related_skill_limit`）并收束介点岗位边界（`jobs_per_skill`）进行数据稀疏化控制，降低传输负担。
- **高拟真原生流体交互**：全部借助流式响应和 HTML API 完成，运用 140ms ~ 160ms 不等的拟态缓冲位移动画及弹性边界响应方案带来了极为丝滑现代的操作物理沉浸体验感。

**代码贡献**：源代码合计约 1150+ 行。其中前端主面板架构、渲染逻辑及艺术 CSS 动画（`woodle_ai_home.html`）约计 900+ 行，后端微型 HTTP Web API 提供程序核心架构模块（`serve.py`）约计 150 行，图结构抽取采样逻辑及映射方法（`graph_query.py`）约计 100 行。
