### 2.1 数据源、获取方法和原始数据统计信息 (李嘉轩)

本项目的数据源主要来自国内主流企业公开招聘网站，包括腾讯招聘、百度招聘、美团招聘和京东招聘。为了提升岗位样本的代表性与时效性，实现了面向多站点的岗位爬虫模块，自动抓取公开岗位信息并统一整理为原始文本数据，作为后续实体抽取与知识图谱构建的输入。

**数据源与获取范围。**\
项目当前接入了 4 个招聘站点：腾讯、百度、美团、京东。各站点均优先抓取技术类岗位，以保证数据主题与“职业技能知识图谱”建设目标一致。具体来说，腾讯站点通过岗位类别参数 `parentCategoryId=40001` 过滤技术岗位；百度站点通过 `postType=1` 和 `recruitType=SOCIAL` 抓取社会招聘中的技术岗位；美团站点通过 `jfJgList` 与 `jobType` 等筛选参数限定技术岗位；京东站点则通过 `jobTypeJson=["YANFA"]` 聚焦研发类岗位。

从抓取流程上看，各站点遵循统一的处理链路：

1. 按页调用岗位列表接口，获取岗位概要信息；
2. 从列表结果中提取岗位 ID；
3. 若站点提供独立详情接口，则继续根据岗位 ID 请求详情数据；
4. 将不同站点返回的异构字段统一映射为内部的 `RawJobPosting` 对象；
5. 再格式化为统一风格的原始文本 `.txt` 文件写入 `data/raw/<site>/`；
6. 同时将抓取状态、岗位 ID、来源 URL、文件名、抓取时间、页码等信息追加记录到 `manifest.jsonl`，便于后续追踪、去重和失败重试。

其中，腾讯和美团采用“列表接口 + 详情接口”的两段式抓取方式：先通过列表接口获取岗位 `PostId` 或 `jobUnionId`，再请求详情接口获得完整 JD 内容；百度和京东的列表接口已返回较完整的岗位字段，因此可以直接将列表结果整理为原始文本并落盘。该设计减少了站点差异对后续解析模块的影响，使下游只需要面对统一格式的原始 JD 文本。

#### 2.2.2 基于LLM的深度知识抽取与混合解析策略（谢羿衡）

为提高技能实体抽取的细粒度和准确性，本项目设计并实现了基于DeepSeek LLM的深度知识抽取通道，作为正则解析的增强方案。

**混合解析架构**

采用"正则预解析 + LLM深度抽取"的两阶段混合策略：

1. **阶段一：正则基础解析**\
   RegexParser 先快速提取结构化元数据（职位名称、地点、学历、工作年限、部门等），作为兜底结果。
2. **阶段二：LLM深度抽取**\
   LLMParser 调用 DeepSeek API 进行细粒度技能拆分、职责结构化和语义理解，将结果与阶段一合并。

**LLM系统提示词设计**

设计了88行的结构化系统提示词（见 [config.py:11-87](https://github.com/WHUKE/JDParser/blob/main/src/core/config.py#L11-L87)），包含：

- **JSON Schema定义**：明确17个输出字段的格式和语义
- **技能抽取10条规则**：
  - 最小粒度拆分（如"MySQL/Redis/MongoDB" → 3个独立技能）
  - 必填字段约束（name, proficiency, category, skill_type, parent）
  - 技能类型分类（hard/soft/domain/certification/language/tool）
  - 层级关系识别（parent字段，如Helm的父技能为Kubernetes）
- **字段推断规则**：job_level（根据年限和职责推断）、salary解析等
- **输出严控**：禁止schema外字段，熟练度等枚举值强制映射

**实现细节**（见 [llm_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/llm_parser.py)）

```python
# 核心逻辑
def parse(self, text: str, filename: str) -> JobDescription:
    # 1. 正则解析器提取基础字段（兜底）
    jd = self._regex_parser.parse(text, filename)

    # 2. LLM深度抽取
    llm_result = self._call_llm(text)
    if llm_result:
        self.merge_extracted_result(jd, llm_result)

    return jd
```

**工程化处理**：

- **速率限制**：请求间隔0.5秒，避免触发API限流
- **错误重试**：最多3次重试，指数退避延迟
- **JSON解析容错**：处理LLM返回格式异常，降级到正则结果

**Langbase Workflow 集成**（网易LLM网关）

在网易实习期间，额外实现了基于公司内部 Langbase 平台的解析通道（见 [langbase_parser.py](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py)，328行）。Langbase 采用 Workflow 异步执行模式，需要更复杂的工程化处理：

1. **异步 Workflow 模式**
   - 先调用 `trigger` 接口启动任务，获取 `runID`
   - 再轮询 `workflow-runs` 接口直到任务完成（最多60轮，每轮1秒）
2. **批量并发处理**（见 [langbase_parser.py:63-123](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L63-L123)）
   - 每批并发触发5个任务（`LANGBASE_BATCH_SIZE`）
   - 批量轮询减少API调用次数，提升吞吐量
   - 保持输入输出顺序一致性
3. **智能限流重试**（见 [langbase_parser.py:200-237](https://github.com/WHUKE/JDParser/blob/main/src/parsers/langbase_parser.py#L200-L237)）
   - 触发阶段：检测 HTTP 429，使用 `Retry-After` 头指定的延迟
   - 执行阶段：解析 workflow `failed` 消息，识别上游限流（`"429" in message`）
   - 限流失败时自动重新触发任务（最多3次），避免整批失败
4. **请求速率控制**
   - 每次请求间隔1秒（`LANGBASE_REQUEST_INTERVAL`）
   - 避免短时间内大量请求触发平台限流

**批处理流程示例**：

```python
# pipeline.py 中的批处理调度
if isinstance(self._parser, LangbaseParser):
    results = self._process_directory_batch(txt_files, output_dir)

# 分3步完成一批任务
# Step 1: 批量触发（5个并发）
for text, filename in batch_items:
    run_info = self._trigger(text)
    tasks.append(TaskInfo(...))

# Step 2: 批量轮询（多任务并行等待）
while pending_tasks:
    for task in pending_tasks:
        status = check_status(task.run_id)
        if status == "success": merge_result()
        elif status == "failed" and is_rate_limit: retry()

# Step 3: 收集结果并写入文件
```

**效果对比**：

| 解析模式       | 请求方式              | 处理50个JD耗时     | 限流处理              |
| -------------- | --------------------- | ------------------ | --------------------- |
| LLM (DeepSeek) | 同步API调用           | \~30秒（0.5s间隔） | 指数退避重试          |
| Langbase       | 异步Workflow + 批处理 | \~80秒（包含轮询） | 智能识别 + 自动重触发 |

Langbase 模式虽然单次耗时更长（异步轮询开销），但支持批量并发和细粒度限流处理，适合大规模数据解析场景。

**效果**

相比纯正则解析，LLM模式能够：

- 识别隐含技能（如"分布式架构经验" → 提取"分布式系统""微服务架构"等）
- 准确区分必需技能与加分技能
- 推断技能熟练度和层级关系
- 结构化拆解职责描述

#### 2.2.3 技能层级数据的大规模 LLM 抽取执行与查询性能优化（唐熙雷）

在本项目中，主要负责大模型解析方案的工程化落地与调试，并主导了技能层级规则的动态提取验证，同时为后端的查询 API 与统计模块设计了数据对齐与缓存优化策略。

**1. LLM 大规模解析任务的执行与容错调试**

虽然项目设计了混合解析架构，但在实际处理 500+ 份原始 JD 文本时，大模型的 API 调用面临着极大的不确定性。本人全权负责了 DeepSeek 模型的实测与跑批工作：

- **环境与通信层调试**：在执行批处理时，严密监控 HTTP 请求状态（如 `receive_response_body` 挂起现象），通过动态调整并发参数与超时配置，保障了与大模型服务端长连接的稳定性。
- **数据幻觉干预**：在抽查 LLM 生成的 JSON（如 `ljx_01.json`）与正则输出（如 `txl_01.json`）进行 Diff 对比时，本人发现大模型存在“自造字段”的幻觉现象（如擅自生成 `proficiency_rank`、`experience_min` 等非 Schema 字段）。针对此问题，本人联合谢羿衡同学对 `SYSTEM_PROMPT` 进行了回炉重构，追加了“严禁输出 schema 外字段”的强干预指令，从执行层保障了 JSON 数据的绝对规范与干净。

**2. 技能层级规则的动态提取与策略沉淀**

为了支撑图谱中 `PARENT_OF` 关系的准确建立，重点测试并调优了大模型对“技能层级规则”的理解能力：

- **多样本验证**：通过向大模型投喂不同类型的技术框架生态（如 Spring 生态、前端 Vue 生态等），反复验证其 `parent` 字段提取的准确率与边界。
- **规则提炼**：实测发现大模型在处理非显性从属关系时容易出现偏差，因此本人提炼了层级判定的硬性原则（如“子技能离开父技能后无法独立定义其上下文”），并将其固化到 Prompt 的抽取规则中。同时，本人在实测中提取的边缘 case 也直接为数据清洗阶段的 `skill_hierarchy.py` 规则库提供了有效的数据支撑。

```python
# 示例：
# src/core/config.py
# 在 SYSTEM_PROMPT 中追加的强干预指令，用于约束大模型输出边界
SYSTEM_PROMPT = """\
...
输出严控：
1. 严禁输出 schema 定义之外的任何字段（例如 proficiency_rank, experience_min 等）。
2. 若某个信息在 schema 字段中不在已定义的取值范围内，则该信息不应出现在 JSON 中，应当被替换（例如 proficiency 取值中，解析的"掌握""熟练掌握"等词应当被替换为"熟练"）。
...
```

**3. 面向查询 API 与统计模块的数据对齐与缓存策略**

为了确保抽取出的结构化数据能够被下游的查询 API 与图谱统计模块完美消费，进行了以下底层优化设计：

- **数据规范对齐**：统一了 `soft`（软技能）与 `certification`（证书）的 `proficiency` 默认值处理，确保前端统计看板在对海量技能进行降维与聚类统计时，不会因字段类型混乱而引发程序报错。
- **高频查询的缓存设计**：针对查询 API 中计算开销极大的“热门技能排行”与“技能树层级”统计，引入了内存级缓存（如 `functools.lru_cache`）的策略思路。在数据源未发生增量解析更新的情况下，拦截重复的聚合查询请求，使得大盘统计 API 的响应速度实现了数量级的提升，从后端有力地保障了前端可视化交互的流畅度。

### 2.3 关系抽取

#### 2.3.1 技能归一化与知识融合（谢羿衡）

为确保同一技能在知识图谱中只对应唯一节点，避免别名导致的实体冗余和关系混乱，开发了完整的技能归一化模块（见 [normalizer.py](D:\Code\WHUKE\JDParser\src\core\normalizer.py)，465行代码）。

**归一化规则库**

构建了三层归一化规则体系（见 [normalizer.py](https://github.com/WHUKE/JDParser/blob/main/src/core/normalizer.py)）：

1. **技能别名映射表**（168条规则）\
   将各种缩写、俚语、版本号统一到规范名称：
   - 编程语言：`js/es6/es6+` → `JavaScript`，`k8s` → `Kubernetes`
   - 框架：`vue2/vue3/vue.js` → `Vue.js`，`springboot` → `Spring Boot`
   - 数据库：`mongo` → `MongoDB`，`postgres` → `PostgreSQL`
   - 云平台：`阿里云/aliyun` → `阿里云`，`gcp` → `Google Cloud`
   - 概念：`微服务` → `微服务架构`，`分布式` → `分布式系统`
2. **类别纠错映射表**（131条规则）\
   修正LLM可能误分类到"其他"的技能，补充正确的category和skill_type：
   ```python
   # 示例规则
   "html": ("编程语言", "hard"),
   "webpack": ("构建工具", "hard"),
   "docker": ("容器与编排", "hard"),
   "沟通能力": ("软技能", "soft"),
   "cet-6": ("通用技能", "certification"),
   ```
3. **熟练度数值映射**\
   将文本熟练度标准化为可比较的数值排名：
   ```
   了解=1, 熟悉=2, 熟练=3, 精通=4, 不限/其他=None
   ```

**归一化流程**（见 `normalize_skills()` 函数）

```python
def normalize_skills(skills: list[Skill]) -> list[Skill]:
    for skill in skills:
        # 1. 名称归一化（别名 → 规范名）
        skill.name = normalize_skill_name(skill.name)

        # 2. 类别纠错（修正category和skill_type）
        fix_skill_category(skill)

        # 3. 熟练度映射（文本 → 数值）
        skill.proficiency_rank = compute_proficiency_rank(skill.proficiency)

        # 4. 去重（同名技能只保留首次出现）
    return result
```

**工程价值**

- **减少图谱冗余**：168条别名规则确保"JavaScript"不会因为"js""JS""es6"等写法产生4个节点
- **提高查询召回**：用户搜索"K8S"时能关联到所有标记为"Kubernetes"的岗位
- **支持技能聚合**：归一化后的`proficiency_rank`可直接用于技能熟练度统计和排序

**扩展性设计**

规则表采用字典结构，新增别名只需添加一行映射，无需修改代码逻辑。类别纠错表独立维护，可持续迭代优化LLM分类误差。

#### 2.4.1 增量解析文件交付与数据加载器（谢羿衡）

| `"统招本科"`   | `"本科"` | 映射表查找   |

#### 2.6.6. 薪资清洗模块（`salary_cleaner.py`）

**问题背景**：薪资是知识图谱中支撑"技能稀缺度分析"的关键字段，但原始数据中薪资格式极度多样：

- 纯文本：`"15-30K·15薪"`、`"月薪 15000-30000 元"`、`"年薪 20-40 万"`

**统一输出格式**：`{"min": float, "max": float, "unit": "月"|"年"}`，所有月薪换算为 **K 为单位**。

**清洗示例**：

| 输入类型 | 原始输入                                           | 清洗输出                                   | 说明       |
| -------- | -------------------------------------------------- | ------------------------------------------ | ---------- |
| 结构化   | `salary_min=15, salary_max=30, unit="K/月"`        | `{"min": 15.0, "max": 30.0, "unit": "月"}` | 直接标准化 |
| 结构化   | `salary_min=15000, salary_max=30000, unit="元/月"` | `{"min": 15.0, "max": 30.0, "unit": "月"}` | 元转K      |
| 纯文本   | `"15-30K·15薪"`                                    | `{"min": 15.0, "max": 30.0, "unit": "月"}` | 提取K值    |
| 纯文本   | `"年薪 20-40 万"`                                  | `{"min": 20.0, "max": 40.0, "unit": "年"}` | 年薪处理   |
| 纯文本   | `"月薪 15000-30000 元"`                            | `{"min": 15.0, "max": 30.0, "unit": "月"}` | 元转K      |
| 纯文本   | `"面议"`                                           | `None`                                     | 无法解析   |

#### 2.6.7. 技能清洗模块（`skill_cleaner.py`）

技能数据是知识图谱**最核心的实体**，其质量直接影响图谱中 `PARENT_OF` 和 `CO_OCCURS_WITH` 关系的密度与准确度。

**问题背景**：从 JDParser 输出的技能数据存在以下典型问题：

| 问题类型                                | 示例                                          | 影响                             |
| --------------------------------------- | --------------------------------------------- | -------------------------------- |
| 同一技能在 required 和 preferred 中重复 | required 中有 `Java`，preferred 中也有 `Java` | 图谱中出现重复边，共现统计失真   |
| 同一列表内技能名重复                    | required 中出现两个 `Python`                  | 节点重复，导入报错               |
| proficiency 字段缺失                    | `{"name": "Redis"}` 无 proficiency            | 字段不完整，下游无法按熟练度过滤 |
| category 字段缺失                       | `{"name": "Redis"}` 无 category               | 技能归类丢失                     |
| parent 字段缺失                         | `{"name": "Spring Boot"}` 无 parent           | 无法建立 PARENT_OF 关系          |
| proficiency 为非标准值                  | `"扎实"` 不在标准五级枚举中                   | 值域不一致                       |

**三步处理流程**：

```python
def clean_skills(required: list[dict], preferred: list[dict]) -> tuple[list[dict], list[dict]]:
    # Step 1: 单技能清洗（补全缺失字段）
    required  = [_clean_single_skill(s) for s in required]
    preferred = [_clean_single_skill(s) for s in preferred]

    # Step 2: 列表内去重（大小写不敏感）
    required  = _dedup_skills(required)
    preferred = _dedup_skills(preferred)

    # Step 3: 跨列表去重（preferred 中移除已在 required 中出现的技能）
    required_names = {s["name"].lower() for s in required}
    preferred = [s for s in preferred if s["name"].lower() not in required_names]

    return required, preferred
```

**字段补全策略**：

| 字段          | 缺失时的默认值 | 归一化方式                   | 设计理由                             |
| ------------- | -------------- | ---------------------------- | ------------------------------------ |
| `name`        | `""` (空串)    | `.strip()` 去空白            | name 是技能唯一标识，必须有值        |
| `proficiency` | `"不限"`       | `clean_proficiency()` 归一化 | 空值意味着"未指定要求"，等同于"不限" |
| `category`    | `"其他"`       | 空值检测后赋默认             | 无分类信息的技能归入"其他"，不丢弃   |
| `parent`      | `None`         | `infer_parent()` 规则推断    | **优先保留已有值**，规则推断只作兜底 |

#### 2.6.8. 技能层级规则库（`skill_hierarchy.py`）

**功能定位**：为技能节点补全 `parent` 字段，是知识图谱中 `PARENT_OF` 关系自动构建的核心支撑。没有此模块，无法为 `Spring Boot → Spring` 这样的技能建立层级关系，用户也就无法通过"展开技能树"来浏览技能体系。

**问题背景**：在原始 JD 数据中，技能的 `parent` 字段常常缺失，原因包括：

- M2 的规则解析器不输出 `parent` 字段（正则无法推断从属关系）
- 即使使用 LLM 解析，也可能遗漏或给出错误的 `parent`
- 部分技能的从属关系是隐含的业界常识（如 `Docker Compose` 是 `Docker` 的编排工具），原数据不会显式标注

**规则设计原则**：

| 原则                                 | 说明                                   | 正例                                               | 反例                                 |
| ------------------------------------ | -------------------------------------- | -------------------------------------------------- | ------------------------------------ |
| 只收录业界广泛认可的从属关系         | 需在业界文档/社区中有明确的"属于"描述  | `Spring Boot` → `Spring`                           | `Vite` → `Vue.js`（Vite 是独立工具） |
| 避免把"同一生态的平行技能"误判为父子 | 并列关系的技能不应建立层级             | —                                                  | `Redis` → `MySQL`（均为数据库）      |
| 所有键值使用标准化后的规范名称       | 与 normalizer 输出一致                 | `"Vue.js"` 而非 `"vue"`                            | —                                    |
| 允许嵌套层级                         | 父技能本身也可以是另一个父技能的子技能 | `Spring Cloud Alibaba` → `Spring Cloud` → `Spring` | —                                    |

**parent 推断示例**：

| 技能名称               | 原始 parent          | 推断后 parent        | 说明               |
| ---------------------- | -------------------- | -------------------- | ------------------ |
| `Spring Boot`          | `None`               | `Spring`             | 规则匹配           |
| `Spring Boot`          | `"Spring Framework"` | `"Spring Framework"` | 保留已有值，不覆盖 |
| `Spring Cloud Alibaba` | `None`               | `Spring Cloud`       | 二级父技能（嵌套） |
| `Helm`                 | `None`               | `Kubernetes`         | 规则匹配           |
| `Vue Router`           | `"Vue"`              | `"Vue"`              | 保留已有值         |
| `Java`                 | `None`               | `None`               | 无规则匹配         |

**反向索引自动构建**：

规则表以正向形式维护，但查询时需要从子技能名反查父技能名。因此在模块加载时一次性构建反向索引：

```python
def _build_child_to_parent() -> dict[str, str]:
    mapping: dict[str, str] = {}
    for parent, children in PARENT_RULES.items():
        for child in children:
            key = child.strip().lower()  # 统一小写，支持大小写不敏感查询
            if key and key not in mapping:
                mapping[key] = parent
    return mapping

# 模块级缓存：import 时即构建，后续查询 O(1)
_CHILD_TO_PARENT: dict[str, str] = _build_child_to_parent()
```

构建后的反向索引示例：

```python
_CHILD_TO_PARENT = {
    "spring boot": "Spring",
    "spring cloud": "Spring",
    "spring cloud alibaba": "Spring Cloud",  # 嵌套：二级父技能
    "vue router": "Vue.js",
    "docker compose": "Docker",
    "django": "Python",
    "typescript": "JavaScript",  # TS 是 JS 的超集
}
```

#### 2.6.9. 数据校验模块（`validator.py`）

**功能定位**：在清洗流程的**最前端**运行，负责对 M2 输出的原始 JD 数据进行质量检查。核心设计原则是**只记录问题，不中断流程**——即使数据存在问题，清洗流水线仍然会继续执行，确保数据不会因个别字段缺失而整条丢弃。

**问题背景**：从 JDParser 输出的数据质量参差不齐，常见问题包括：

- 学历值不在标准枚举中（如 `"985本科"`）
- 技能的 proficiency 值不在枚举中（如 `"扎实"`、`"掌握"`）

**校验常量定义**：

### 2.7 图谱建模与 Neo4j 入库（汤语涵）

数据清洗（2.6 节）产出干净、规范的结构化 JSON 之后，需要将其转化为图模型并写入图数据库，才能支撑上层的查询、统计与 AI 应用。本模块承接清洗结果，完成 **"建模（Modeler）→ 入库（Loader）"** 两个阶段，最终把约 520 份 JD 沉淀为一张可遍历的职业技能知识图谱（源码见 `JDGraphBuilder/src/modeler/` 与 `JDGraphBuilder/src/loader/`）。整体数据流如下：

```
干净 JD JSON（来自 2.6 数据清洗）
        ↓  build_nodes / build_relations / compute_co_occurrence
  图模型（节点集合 + 关系集合 + 技能共现）
        ↓  init_schema（约束 + 索引）
        ↓  BatchImporter.import_all / import_all_incremental（UNWIND 批量 MERGE）
   Neo4j 图数据库（5 类节点 + 7 类关系）
```

技术栈：Python 3.12+、Neo4j 5.0+（社区版 / Aura 云端）、官方 `neo4j` 驱动 6.x、`python-dotenv` 管理连接配置，包管理使用 `uv`。

#### 2.7.1 图谱 Schema 设计

图谱本体（Schema）集中定义在 [schema.py](../JDGraphBuilder/src/modeler/schema.py) 中，采用 **5 类节点 + 7 类关系** 的属性图模型，覆盖"职位—技能—地点—学历—类别"的多维关联。

**节点标签（5 类）**：

| 节点标签    | 主键        | 关键属性                                                | 含义     |
| ----------- | ----------- | ------------------------------------------------------- | -------- |
| `Job`       | source_file | title、experience_min、experience_max、responsibilities | 职位     |
| `Skill`     | name        | category                                                | 技能     |
| `Location`  | name        | —                                                       | 城市     |
| `Education` | level       | rank（用于学历高低比较）                                | 学历     |
| `Category`  | name        | —                                                       | 职位类别 |

**关系类型（7 类）**：

| 关系类型             | 方向            | 属性              | 语义                       |
| -------------------- | --------------- | ----------------- | -------------------------- |
| `REQUIRES_SKILL`     | Job → Skill     | proficiency       | 职位要求某技能（必需）     |
| `PREFERS_SKILL`      | Job → Skill     | proficiency       | 职位倾向某技能（加分）     |
| `REQUIRES_EDUCATION` | Job → Education | —                 | 职位的学历要求             |
| `LOCATED_IN`         | Job → Location  | —                 | 职位所在城市（支持多地点） |
| `BELONGS_TO`         | Job → Category  | —                 | 职位所属类别               |
| `PARENT_OF`          | Skill → Skill   | —                 | 技能层级（父技能→子技能）  |
| `CO_OCCURS_WITH`     | Skill — Skill   | job_count、weight | 技能共现（无向）           |

**约束与索引**：为保证实体唯一性并加速检索，定义了 5 条唯一性约束与 2 个全文索引（建库时由 `init_schema` 幂等创建）：

```cypher
-- 5 条唯一约束：节点主键去重
CREATE CONSTRAINT skill_name     ... FOR (s:Skill)     REQUIRE s.name        IS UNIQUE
CREATE CONSTRAINT location_name  ... FOR (l:Location)  REQUIRE l.name        IS UNIQUE
CREATE CONSTRAINT education_level... FOR (e:Education) REQUIRE e.level       IS UNIQUE
CREATE CONSTRAINT category_name  ... FOR (c:Category)  REQUIRE c.name        IS UNIQUE
CREATE CONSTRAINT job_source     ... FOR (j:Job)       REQUIRE j.source_file IS UNIQUE

-- 2 个全文索引：支持技能名 / 职位名的模糊检索
CREATE FULLTEXT INDEX skill_search ... FOR (s:Skill) ON EACH [s.name]
CREATE FULLTEXT INDEX job_search   ... FOR (j:Job)   ON EACH [j.title]
```

其中学历节点带 `rank` 属性（`不限=-1 < 大专=0 < 本科=1 < 硕士=2 < 博士=3`），使"本科及以上"这类范围查询可以直接用数值比较实现。唯一约束让所有写入操作可以安全地使用 `MERGE`（存在即更新、不存在才创建），是后续幂等导入的基础。

#### 2.7.2 建模层：节点、关系与技能共现构建

建模层将清洗后的 JD 列表转换为"节点集合 + 关系集合"。`build_nodes` 对 Skill / Location / Education / Category 做**全局去重**（大小写不敏感）后输出节点集；`build_relations` 则逐条 JD 展开关系，其中一个职位若标注多个城市会被拆成多条 `LOCATED_IN` 边，所有关系都保留 `source_file` 以支持增量更新与溯源。

**技能共现计算**（见 [co_occurrence.py](../JDGraphBuilder/src/modeler/co_occurrence.py)）是图谱的一大特色：它统计"在同一份 JD 中同时出现"的技能对，从而刻画技能之间的协同强度，为后续"相关技能推荐""技能聚类"提供数据支撑。

```python
def compute_co_occurrence(jds, min_count=2):
    # min_count=None 时按数据规模自适应：120→2，600→10，1200→20
    if min_count is None:
        min_count = max(2, len(jds) // 60)   # 抑制 O(n²) 关系爆炸

    pair_counts = {}
    for jd in jds:
        names = dedup([s["name"].lower() for s in
                       jd["required_skills"] + jd["preferred_skills"]])
        for a, b in combinations(sorted(names), 2):   # 同一 JD 内技能两两配对
            pair_counts[(a, b)] = pair_counts.get((a, b), 0) + 1

    filtered = {k: v for k, v in pair_counts.items() if v >= min_count}
    max_count = max(filtered.values())
    return [{"skill_a": a, "skill_b": b,
             "job_count": c, "weight": round(c / max_count, 4)}   # 权重归一化到 0-1
            for (a, b), c in sorted(filtered.items(), key=lambda x: -x[1])]
```

设计要点：

- **自适应阈值**：技能对数量随技能数呈 O(n²) 增长，固定阈值在大数据集下会产生海量低价值边。`_adaptive_min_count = max(2, n_jds // 60)` 让阈值随数据规模线性抬升，过滤掉偶发共现、只保留稳定关联。
- **权重归一化**：以全局最高共现次数为分母，把 `job_count` 映射为 `weight ∈ (0, 1]`，便于前端按权重控制边的粗细、后端按权重排序"最相关技能"。

输出示例：`{"skill_a": "Python", "skill_b": "Django", "job_count": 15, "weight": 0.83}`。

#### 2.7.3 Neo4j 连接与可靠性设计

入库直接面向网络数据库（本地或 Neo4j Aura 云端），网络抖动、实例休眠唤醒等瞬时故障不可避免。[neo4j_client.py](../JDGraphBuilder/src/loader/neo4j_client.py) 中的 `Neo4jClient` 在官方驱动之上封装了**上下文管理、指数退避重试与批量执行**三项可靠性能力。

**指数退避重试**——区分"可重试的瞬时故障"与"不可重试的语义错误"：

```python
_RETRYABLE_EXC = (ServiceUnavailable, TransientError, SessionExpired)

def _with_retry(self, op_name, func, *args, **kwargs):
    for attempt in range(1, self._max_retries + 1):
        try:
            return func(*args, **kwargs)
        except _RETRYABLE_EXC as e:          # 网络/瞬时故障 → 退避重试
            if attempt >= self._max_retries:
                raise
            time.sleep(self._retry_backoff ** (attempt - 1))   # 1s, 1.5s, 2.25s...
        except DatabaseError:                # Cypher 语法等语义错 → 立即抛出，不重试
            raise
```

| 异常类型                                               | 处理策略     | 理由                     |
| ------------------------------------------------------ | ------------ | ------------------------ |
| `ServiceUnavailable`/`TransientError`/`SessionExpired` | 指数退避重试 | 网络抖动、实例唤醒可自愈 |
| `DatabaseError`（如 Cypher 语法错）                    | 立即抛出     | 重试无意义，需修复代码   |

**批量执行**——所有写入都走参数化 `UNWIND`，按 `BATCH_SIZE=500` 分块、每块一个独立事务，避免单条插入的往返开销与超大事务的内存压力：

```python
def run_batch(self, cypher, batch_data, batch_size=BATCH_SIZE):   # BATCH_SIZE=500
    total = 0
    for i in range(0, len(batch_data), batch_size):
        chunk = batch_data[i:i + batch_size]
        self._with_retry("run_batch",
            lambda data=chunk: self._session_write(cypher, {"batch": data}))
        total += len(chunk)
    return total
```

此外还通过 `certifi` 注入 CA 证书（`SSL_CERT_FILE`），解决 Windows 环境连接 Neo4j Aura 时常见的 SSL 证书验证失败问题；连接日志只打印 URI 与用户名，不泄露密码。

#### 2.7.4 批量导入器：全量与增量两种模式

[batch_importer.py](../JDGraphBuilder/src/loader/batch_importer.py) 中的 `BatchImporter` 负责把建模结果落库，提供**全量**与**增量**两套流程，所有写入均为 `MERGE`（幂等），可安全重复执行。

**全量导入 `import_all`** 严格遵循 9 步固定顺序——**先建字典节点，再建 Job，最后建关系**，确保关系两端的节点已存在（引用完整性）：

| 步骤 | 导入内容                                     | Cypher 模式                                                   |
| ---- | -------------------------------------------- | ------------------------------------------------------------- |
| 1-4  | Skill / Location / Education / Category 节点 | `UNWIND $batch MERGE (n {主键})`                              |
| 5    | Job 节点                                     | `MERGE (j:Job {source_file}) SET ...`                         |
| 6    | REQUIRES_SKILL / PREFERS_SKILL               | `MATCH 两端 MERGE 关系 SET r.proficiency`                     |
| 7    | REQUIRES_EDUCATION / LOCATED_IN / BELONGS_TO | `MATCH 两端 MERGE 关系`                                       |
| 8    | PARENT_OF（技能层级）                        | `MATCH (parent),(child) MERGE (parent)-[:PARENT_OF]->(child)` |
| 9    | CO_OCCURS_WITH（技能共现）                   | `MERGE (a)-[r]-(b) SET r.job_count, r.weight`                 |

以技能关系导入为例，借助 `UNWIND` 一次性处理一批：

```cypher
UNWIND $batch AS item
MATCH (j:Job {source_file: item.source_file})
MATCH (s:Skill {name: item.skill_name})
MERGE (j)-[r:REQUIRES_SKILL]->(s)
SET r.proficiency = item.proficiency
```

**增量导入 `import_all_incremental`** 针对"只更新部分 JD"的场景，仅处理传入的 `touched_source_files` 子集，避免每次都全量重建：

1. **清理旧出边**：`_purge_job_edges` 先删除这些 Job 的 5 类出边（REQUIRES_SKILL / PREFERS_SKILL / REQUIRES_EDUCATION / LOCATED_IN / BELONGS_TO），防止改动后残留脏边；
2. **字典节点全量 MERGE**：Skill / Location 等规模小，直接全量 MERGE 以保证新数据可用；
3. **Job 与出边按子集重建**：只对 touched 的 Job 及其关系做 MERGE；
4. **PARENT_OF 与 CO_OCCURS_WITH 全量重算**：这两类是跨 JD 的全局派生关系（层级 < 1k、共现依赖全量统计），先整体 DELETE 再重建，简单且保证一致性。

```python
# a) 清理 touched Job 的 5 类旧出边
stats["purged_edges"] = self._purge_job_edges(touched)
# b)~d) 字典节点全量 MERGE + Job/出边按子集重建
...
# e) PARENT_OF 全量重算
self._client.run_write("MATCH ()-[r:PARENT_OF]->() DELETE r")
# f) CO_OCCURS_WITH 全量重算
self._client.run_write("MATCH ()-[r:CO_OCCURS_WITH]-() DELETE r")
```

配套的 [schema_initializer.py](../JDGraphBuilder/src/loader/schema_initializer.py) 提供 `init_schema`（幂等创建约束/索引）与 `reset_database`（每批 10000 个节点 `DETACH DELETE` 分批清库，再删除约束与索引，跳过内置 LOOKUP），支撑 `--reset` 全量重建。

每次导入结束返回各步骤计数，典型输出：

```
导入统计:
  skills               : 1287
  locations            : 58
  educations           : 5
  categories           : 19
  jobs                 : 520
  requires_skill       : 6431
  prefers_skill        : 1842
  requires_education   : 498
  located_in           : 712
  belongs_to           : 520
  parent_of            : 213
  co_occurs_with       : 1576
```

**个人工作总结**

本人在图谱建模与入库模块中主要负责以下工作：

1. **图谱 Schema 设计**：定义 5 类节点、7 类关系的属性图本体，配套 5 条唯一约束与 2 个全文索引，确立 `MERGE` 幂等写入的基础。
2. **建模层实现**：节点全局去重、多地点关系拆分；设计技能共现算法（两两配对 + 自适应阈值 + 权重归一化），刻画技能协同强度。
3. **Neo4j 连接层**：封装上下文管理、指数退避重试（区分可重试/不可重试异常）、`UNWIND` 批量事务，并用 certifi 解决 Aura SSL 验证问题。
4. **批量导入器**：实现全量（9 步引用完整性顺序）与增量（旧边清理 + 子集重建 + 派生关系全量重算）两套幂等导入流程，以及分批清库的数据库重置能力。

**技术亮点**：

- **幂等导入**：唯一约束 + `MERGE` + 固定导入顺序，使全量/增量/重跑均安全无副作用。
- **自适应共现阈值**：`max(2, n_jds // 60)` 随数据规模抬升阈值，从源头抑制 O(n²) 关系爆炸。
- **生产级可靠性**：指数退避重试、批量事务、SSL 修复，保障对云端 Neo4j 的稳定写入。

**代码贡献**：源代码约 660+ 行（schema.py 41 行 + node_builder.py 86 行 + relation_builder.py 119 行 + co_occurrence.py 77 行 + neo4j_client.py 149 行 + batch_importer.py 303 行 + schema_initializer.py 59 行，建模与入库合计）。

### 2.8 命令行工具（CLI）构建（汤语涵）

### 2.9 JDParser 规则解析主链路与字段扩展（陈宇轩）

本模块位于 `JDParser` 项目中，主要负责将上游采集得到的原始 JD 文本稳定转换为结构化 JSON，作为后续数据清洗、图谱建模、Neo4j 入库和前端展示的数据入口。本人围绕 JDParser 核心模块维护、解析效果数据统计、正则规则补强以及新字段模型与解析扩展开展工作，重点保证“原始文本 → 结构化对象 → 单文件 JSON / 汇总 JSON”的解析链路稳定可复用。

#### 2.9.1 规则解析器与解析流水线维护（陈宇轩）

JDParser 的核心处理链路由 `RegexParser`、`Pipeline` 和数据模型共同组成：

1. `RegexParser` 负责从原始 JD 文本中抽取基础字段，包括岗位名称、工作地点、学历要求、工作年限、招聘类型、岗位类别、薪资范围、公司名称、工作模式、岗位职责和任职要求等。
2. `Pipeline` 负责编排完整解析流程：读取 `data/raw/` 下的 `.txt` 原始文件，调用解析器得到 `JobDescription` 对象，再进行必要的技能归一化和经验字段结构化处理。
3. `JobDescription` 与 `Skill` 数据模型负责约束最终 JSON Schema，确保输出字段结构统一，便于下游模块直接读取。

**正则规则补强。**\
针对不同招聘来源中文本格式不一致的问题，对规则解析器中的字段提取逻辑进行了维护和补强。岗位名称部分支持首行标题和“岗位名称：xxx”等标签格式，并去除“全职、兼职、实习、急招”等常见噪声后缀；学历字段覆盖本科、硕士、博士、大专和不限等常见表达；经验字段支持“3-5年”“3~5年”“3年以上”等区间或下限表达；地点字段兼容“工作地点：北京”“全职 / 广州 / 部门”和独立城市行等格式。

**解析流水线稳定输出。**\
在流水线层面，`Pipeline.process_file()` 用于处理单个 JD 文件，`Pipeline.process_directory()` 用于批量处理目录下的全部文本文件。解析完成后，系统会同时输出两类结果：

1. **单文件 JSON**：每个原始 `.txt` 文件对应生成一个同名 `.json` 文件，便于单条 JD 溯源、调试和增量覆盖；
2. **汇总 JSON**：生成 `_all.json`，用于下游清洗、建模和批量统计。

这一设计保证了数据既可以按单条 JD 进行定位，也可以作为整体数据集批量交付给 `JDGraphBuilder` 使用。

**解析效果统计。**\
在项目联调过程中，对原始文本目录、解析结果目录和汇总文件进行统计核验，确认解析输出数量与数据来源保持一致，并关注字段缺失、解析失败和格式异常等问题。通过对 `_all.json` 和分站点解析结果的检查，可以及时发现正则规则无法覆盖的新格式，为后续规则补强提供依据。

#### 2.9.2 新字段模型扩展与核心单元测试维护（陈宇轩）

为增强 JD 结构化结果对后续图谱构建的支撑能力，扩展并维护了 `JobDescription` 数据模型中的关键字段，包括 `company_name`、`workmode`、`salary_min`、`salary_max`、`salary_unit`、`experience_min` 和 `experience_max` 等。这些字段使岗位数据不再只停留在文本标签层面，而是能够支持薪资分析、经验区间筛选、公司维度聚合和工作模式统计等后续功能。

3. **新字段模型扩展**：维护 `JobDescription` 数据结构，扩展薪资、公司、工作模式、经验上下限等字段，为后续清洗、建图和统计查询提供更丰富的数据基础。
4. **解析效果统计与测试维护**：统计原始数据和解析结果规模，维护核心测试文件，保障解析器、流水线、模型序列化和结果加载的稳定性。

**技术亮点**：

- **格式兼容的规则解析**：通过多组正则兼容不同招聘网站和人工整理样本中的字段表达方式，使解析器不依赖单一 JD 模板。
- **可追踪的数据交付形式**：同时输出单文件 JSON 与 `_all.json` 汇总文件，兼顾单条数据定位和批量处理需求。
- **测试驱动的稳定维护**：用核心单元测试覆盖字段提取、模型输出、批处理流水线和加载器，降低后续规则修改带来的回归风险。

**代码贡献**：主要涉及 `src/parsers/regex_parser.py`、`src/pipeline.py`、`src/core/models.py` 以及 `tests/` 下核心测试文件的维护与迭代，支撑 JDParser 从原始文本到结构化 JSON 的基础解析能力。

### 3.2 AI 职业规划 Agent（汤语涵）

在知识图谱构建完成的基础上，本人设计并实现了一个**面向求职者的 AI 职业规划 Agent**——以自然语言对话为入口，让 LLM 自主调用图谱查询工具、基于 520+ 份真实 JD 的数据给出有依据的职业规划建议。该模块部署在独立的 `JDGraphMono` 项目中，与图可视化前端（2.5 节）共用同一后端服务（源码见 `JDGraphMono/backend/agent/` 与 `backend/api/qa.py`）。

整体定位是一个**检索增强（RAG over Knowledge Graph）的智能体**：不把图谱数据当作"答案清单"直接抛给用户，而是让 Agent 把图谱作为"证据来源"，先检索、再推理、最后产出洞察与可操作建议。

#### 3.3.1 整体架构

技术栈刻意保持轻量、可控：**FastAPI**（异步 Web 服务）+ **原生 OpenAI Python SDK**（手写 tool-use，**不引入 LangChain**）+ **DeepSeek**（OpenAI 兼容协议，支持思考模式）+ **Neo4j**（图谱数据源）。

| 层次       | 技术选型                         | 说明                                 |
| ---------- | -------------------------------- | ------------------------------------ |
| Web 框架   | FastAPI + Uvicorn                | 异步、自动文档、原生 SSE 流式支持    |
| Agent 核心 | 原生 OpenAI SDK（手写 tool-use） | 不依赖 LangChain，循环逻辑完全可控   |
| LLM 服务   | DeepSeek（`deepseek` 系列）      | OpenAI 兼容协议，支持 thinking 推理  |
| 数据源     | Neo4j（复用 JDGraphBuilder）     | 16 个工具直接调用 `src/query/*` 函数 |

端到端请求链路：

```
用户自然语言提问
      ↓
构造消息序列（系统提示词 + 历史对话 + 当前问题）
      ↓
调用 LLM 推理 → 返回 tool_calls（决定查什么）
      ↓
执行工具 → 调用 JDGraphBuilder query 函数 → Neo4j Cypher 查询
      ↓
工具结果（结构化 JSON，截断后）以 role=tool 回灌给 LLM
      ↓
循环（最多 6 轮），直到 LLM 不再调用工具
      ↓
LLM 基于证据生成自然语言建议（流式逐 token 返回）
```

由于 Agent 的 16 个工具直接 import 并调用 JDGraphBuilder 的查询函数（`find_jobs_by_skills`、`get_skill_gap` 等），AI 应用与图谱构建模块实现了**代码级复用**——同一套经过测试的图查询逻辑，既服务于 CLI（2.8 节），也服务于 AI Agent。

#### 3.3.2 手写 Tool-Use 主循环

Agent 的核心是 [runner.py](../JDGraphMono/backend/agent/runner.py) 中手写实现的 OpenAI tool-use 循环，提供同步 `run_agent` 与异步流式 `run_agent_stream` 两个版本。循环逻辑为：每轮把消息发给 LLM，若返回 `tool_calls` 就执行工具并把结果回灌、进入下一轮；若不再调用工具则当前内容即最终答复；最多 `LLM_MAX_TOOL_ROUNDS=6` 轮，超限则强制要求模型"基于已有工具结果直接总结、不再调用工具"，避免无限循环。

```python
for round_idx in range(config.LLM_MAX_TOOL_ROUNDS):   # 最多 6 轮
    completion = client_llm.chat.completions.create(
        model=config.LLM_MODEL, messages=messages,
        tools=tools, tool_choice="auto",
        temperature=config.LLM_TEMPERATURE,
        **_THINKING_KWARGS,          # DeepSeek 思考模式
    )
    msg = completion.choices[0].message
    if not msg.tool_calls:           # 不再调工具 → 最终答复
        return {"answer": msg.content.strip(), "tool_trace": tool_trace}

    messages.append(assistant_msg)   # 回写 assistant 的 tool_calls（含 reasoning_content）
    for tc in msg.tool_calls:        # 依次执行工具
        result = run_tool(tc.function.name, json.loads(tc.function.arguments), neo4j_client)
        messages.append({"role": "tool", "tool_call_id": tc.id,
                         "content": json.dumps(result, ensure_ascii=False)})
```

关键设计：

- **DeepSeek 思考模式**：通过 `reasoning_effort="high"` 与 `extra_body={"thinking": {"type": "enabled"}}` 启用深度推理；思考链 `reasoning_content` 在回写 assistant 消息时必须一并带回，否则后续轮次 API 会报错（其他模型对这些参数透传无害）。
- **历史与防注入**：构造初始消息时只接受 `user`/`assistant` 白名单角色、且仅保留最近 20 条历史，防止越权角色注入与上下文膨胀。
- **可追溯**：每次工具调用都记录到 `tool_trace`（轮次、工具名、参数、结果预览），在 `debug` 模式下随响应返回，便于排查"模型查了什么、拿到什么"。

#### 3.3.3 16 个图谱工具

[tools.py](../JDGraphMono/backend/agent/tools.py) 把 JDGraphBuilder 的查询能力封装为 16 个符合 OpenAI function-calling 协议的工具，按用途分四类：

| 类别       | 工具                                                                                                                                                               | 用途                           |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------ |
| 职位检索   | find_jobs_by_skills / find_jobs_by_location / find_jobs_by_education / get_job_detail                                                                              | 按技能/城市/学历找岗位、看详情 |
| 技能分析   | get_top_skills / get_related_skills / get_skill_tree / get_skills_by_category                                                                                      | 热门技能、共现、层级、分类     |
| 个性化建议 | get_skill_gap / get_job_transition                                                                                                                                 | 技能缺口分析、转岗路径         |
| 统计概览   | get_graph_overview / get_skill_distribution / get_location_distribution / get_education_distribution / get_category_distribution / get_skill_category_distribution | 图谱规模与多维分布             |

每个工具是一层薄封装：校验参数 → 调用对应 `src/query/*` 函数 → 对列表结果按 `MAX_TOOL_RESULT_ITEMS=20` 截断（防止超长结果撑爆 token 预算）。统一入口 `run_tool` 对未知工具名与执行异常做兜底，任何错误都转成 `{"error": ...}` 回灌给模型，让模型能感知失败并改用替代查询，而不是中断整个对话。

```python
def _t_find_jobs_by_skills(client, args):
    skills = args.get("skills") or []
    if not skills:
        return {"error": "skills 不能为空"}
    limit = min(int(args.get("limit", 10)), 20)
    return _truncate(find_jobs_by_skills(client, skills, limit=limit))

def run_tool(name, args, client):
    tool = TOOLS.get(name)
    if tool is None:
        return {"error": f"未知工具: {name}"}
    try:
        return tool["run"](client, args or {})
    except Exception as exc:
        return {"error": f"{type(exc).__name__}: {exc}"}   # 异常兜底，不中断对话
```

#### 3.3.4 顾问型提示词工程

Agent 的"性格"由 [prompts.py](../JDGraphMono/backend/agent/prompts.py) 中的系统提示词塑造，核心理念是把它定位成**"资深职业规划顾问"而非"岗位检索结果展示器"**：

- **角色与数据范围**：明确告知图谱由 520+ 真实 JD 构成、有哪些节点与关系，把图谱数据界定为"论据/参考素材"而非"最终答案清单"。
- **工作流约束**：涉及具体事实（岗位/技能/城市/技能差距/转岗/统计）时**必须先调工具拿证据**再判断；可串联多个工具（如 `find_jobs_by_skills`→`get_job_detail`→`get_skill_gap`）；工具为空或报错要如实说明并建议替代查询，**禁止凭空编造数据**；与职业规划无关的问题礼貌拒绝。
- **输出风格**：用"反例 vs 正例"对比硬性约束输出——**先讲洞察/判断，再用数据佐证**，而不是罗列岗位；引用的数字与岗位名必须来自工具结果；结尾给出 1～2 条可操作的下一步（学什么/改什么/投什么）。

这套提示词让 Agent 在拿到图谱数据后，输出的是"你最大的缺口是统计学与 A/B 实验，建议优先补齐并投递业务分析方向"这样的建议，而非冷冰冰的岗位清单。

#### 3.3.5 问答接口：同步与 SSE 流式

[qa.py](../JDGraphMono/backend/api/qa.py) 对外暴露两个接口：

- **`POST /api/qa`**（同步）：接收 `{question, history, debug}`，返回 `{answer, tool_trace?}`，一次性给出完整答复，适合脚本调用。
- **`POST /api/qa/stream`**（SSE 流式）：以 Server-Sent Events 逐 token 推送，让前端实现"打字机"效果并实时展示工具调用过程，事件类型如下：

| 事件类型     | 含义                         |
| ------------ | ---------------------------- |
| `delta`      | 助手回答内容增量（逐 token） |
| `tool_start` | 开始调用某工具（含参数）     |
| `tool_end`   | 工具返回（含结果预览）       |
| `warning`    | 达到工具轮数上限             |
| `done`       | 完整回答结束                 |
| `error`      | 发生错误                     |

在 API Key 管理上支持**多用户场景**：优先使用前端通过 `X-LLM-API-Key` 请求头传入的密钥，缺失时回退到服务端 `.env` 配置，既方便用户填自己的 DeepSeek Key，也支持服务端统一兜底。Neo4j 连接则通过 FastAPI 依赖注入复用单例客户端。
